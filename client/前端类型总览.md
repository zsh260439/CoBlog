# 前端类型总览

这份文档汇总当前前端项目里**全部和类型定义有关的内容**，并补充两类说明：

1. 这个类型是做什么的
2. 这个类型里每一个字段是什么意思

这次整理后的结构，已经尽量把原来散落在组件里的类型都收到了 `client/src/types` 下，方便后面接 `NestJS`。

***

## 一、文章内容相关类型

这部分类型主要负责描述：

- 一篇文章在前端完整是什么结构
- 文章表单提交时需要哪些字段
- 共享的帖子结构长什么样

### 1. `client/src/types/article.ts`

这是前端文章系统里最核心的业务类型。

```ts
export interface Article {
  _id: string // 文章唯一标识
  slug: string // 文章路由别名，用于 /article/:slug
  title: string // 文章标题
  content: string // 文章正文 Markdown 内容
  excerpt: string // 文章摘要，用于列表展示
  summary: string // 文章顶部总结，用于详情页摘要卡
  category: string // 分类中文名称，例如“技术”
  categorySlug: string // 分类路由标识，例如 technology
  coverImage?: string // 封面图地址，可选
  createdAt: string // 创建时间
  updatedAt: string // 更新时间
  views: number // 浏览量
  comments: number // 评论数
  likes: number // 点赞数
  wordCount: number // 文章字数
}

export interface ArticleFormData {
  slug?: string // 表单中的文章别名，可选
  title: string // 表单提交的标题
  content: string // 表单提交的正文
  excerpt: string // 表单提交的摘要
  summary?: string // 表单提交的总结说明，可选
  category: string // 表单中的分类中文名
  categorySlug?: string // 表单中的分类别名，可选
  coverImage?: string // 表单中的封面图地址，可选
}
```

### 2. `shared/types/post.ts`

这是项目里共享的帖子结构定义，当前前端可能会参考到。

```ts
export interface Post {
```

## 五、UI 组件类型

这部分是这次重点整理的内容。

现在原来分散在组件里的 `Props`、局部展示结构，已经尽量统一放到了：

- `client/src/types/ui.ts`

### 1. `client/src/types/ui.ts`

```ts
export interface Particle {
  x: number // 首页粒子横坐标
  y: number // 首页粒子纵坐标
  vx: number // 横向速度
  vy: number // 纵向速度
  size: number // 粒子大小
  opacity: number // 粒子透明度
}

export interface MarkdownViewerProps {
  content: string // Markdown 渲染组件接收的正文内容
}

export interface ScrollNarrativeProps {
  sections: NarrativeSection[] // 滚动叙事组件接收的分区数组
}

export interface PageHeroProps {
  eyebrow?: string // 横幅上方的小标题，可选
  title: string // 横幅主标题
  description?: string // 横幅描述说明，可选
  image?: string // 横幅背景图地址，可选
  align?: 'left' | 'center' // 内容对齐方式
  height?: 'medium' | 'large' // 横幅高度规格
  mistHeight?: number // 底部白雾过渡高度，可选
}

export interface PostCardProps {
  post: Article // 卡片对应的文章数据
  featured?: boolean // 是否作为重点文章卡片展示
}

export interface PostTocProps {
  items: MarkdownHeading[] // 目录项数组
  activeId?: string // 当前激活的目录标题 ID
}

export interface BlogListItemProps {
  post: Article // 博客列表项对应的文章数据
}

export interface ProfileSidebarCardProps {
  posts: Article[] // 当前页面可用的文章数组，用于统计文章数和分类数
  ownerName: string // 侧栏卡片显示的站点主人名称
  ownerRole: string // 主人角色说明
  ownerLocation: string // 主人所在地
  imageUrl?: string // 头像图片地址，可选
  actionLabels?: string[] // 底部操作项文本数组，可选
}

export interface StatItem {
  label: string // 统计项名称
  value: string // 统计项值
}

export interface SiteStatsCardProps {
  items?: StatItem[] // 站点统计卡片接收的统计数组，可选
}

export interface GuestbookEntry {
  id: string // 留言唯一标识
  author: string // 留言者昵称
  location: string // 留言者位置
  device: string // 留言设备
  browser: string // 留言浏览器
  content: string // 留言正文
  createdAt: string // 留言创建时间
}
```

补充说明：

- `PageHeroProps` 里的 `eyebrow?: string`
  - `eyebrow`：横幅上方的小标题文字
  - `?`：表示可传可不传
  - `string`：说明它是字符串类型

- `ProfileSidebarCardProps` 里的 `imageUrl?: string`
  - `imageUrl`：头像图片地址
  - `?`：表示这个字段是可选的
  - `string`：说明它是字符串类型

- `SiteStatsCardProps` 里的 `items?: StatItem[]`
  - `items`：站点统计数据数组
  - `?`：表示这个字段是可选的
  - 如果不传，会直接使用站点默认统计配置

也就是说，像 `eyebrow?: string` 这样的写法，本质表示：

- 字段名：`eyebrow`
- 是否必填：不是，属于可选字段
- 字段类型：字符串

## 六、mock 数据辅助类型

这部分类型主要服务于前端本地假数据系统。

```ts
export interface CoverPalette {
  skyTop: string // 封面图顶部颜色
  skyBottom: string // 封面图底部颜色
  glow: string // 发光高亮颜色
  surface: string // 表面主色
  silhouette: string // 剪影颜色
}

export type ArticleSeed = Omit<Article, '_id' | 'updatedAt' | 'wordCount' | 'coverImage'> & {
  coverPalette: CoverPalette // 生成 mock 封面图时使用的配色结构
}
```

补充说明：

- `CoverPalette` 是 mock 封面图的配色描述
- `ArticleSeed` 是文章原始种子数据结构
- 真正页面里使用的是加工后的 `Article`

## 八、现在还剩哪些类型没有必要再抽

目前大部分业务类型已经被抽到了 `client/src/types`。

剩下仍然直接在代码里使用但不需要再抽离的，主要是原生类型：

- `HTMLElement`
- `IntersectionObserver`
- `CanvasRenderingContext2D`

这些属于浏览器或 Vue 提供的基础类型，不属于项目自己的业务模型，所以保留原样即可。

***

## 九、为什么这样整理更适合后面接 NestJS

你后面如果切到 `NestJS`，这种整理方式会明显更舒服，因为：

1. 文章结构已经集中：`Article / ArticleFormData`
2. 分类和站点结构已经集中：`SiteCategory / SiteNavItem / SiteSection`
3. 组件输入也集中到了 `ui.ts`
4. mock 层和真实接口层更容易对齐

后面你要继续演进时，可以进一步拆成：

- `types/content.ts`
- `types/site.ts`
- `types/ui.ts`
- `types/api.ts`

1. 再把这些类型按 `NestJS DTO` 方向拆一次
