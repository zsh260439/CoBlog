## 一、当前项目背景

### 1. 项目定位

`CoBlog` 是一个以博客展示为核心的个人全栈项目，前端采用 `Vue 3 + TypeScript + Vite`，后续后端采用 `NestJS + MongoDB + Mongoose`。

当前前端已具备：

- 博客首页
- 文章详情页
- 分类页
- 归档页
- 关于页
- 留言页

后端当前优先实现：

- 文章列表接口
- 文章详情接口
- 分类文章接口

- 留言列表与留言提交接口

当前学习目标不是“把 NestJS 全部学完”，而是：

## 二、当前阶段对 NestJS 的核心理解

### 1. NestJS 不是完全陌生的新体系

如果已经理解过 `Express` 的基本路由和请求响应，那么 NestJS 可以理解为：

- `Express`：自己组织后端工程
- `NestJS`：框架帮你规定更清晰的工程结构

其核心价值不在于“换了一门语言”，而在于：

- 模块化
- 依赖注入
- 控制器与服务分层
- DTO 校验
- 更适合中大型项目维护

### 2. 当前最核心的三个概念

#### Module

模块负责组织一个功能块。

例如：

- `ArticlesModule`
- `MessageModule`

它的作用是把：

- `controller`
- `service`
- `mongoose model`

组装到一起。

#### Controller

控制器负责接收请求、拿参数、返回响应。

它更像“接口入口层”。

不应该在这里写大量复杂业务逻辑。

#### Service

服务层负责真正的数据处理和业务逻辑。

例如：

- 查询文章
- 根据 slug 查详情
- 按年份分组生成归档
- 创建新文章并计算字数

## 三、前后端联调过程中最关键的认识

### 1. 页面路由 和 后端接口路由 不是一回事

需要明确区分：

- 前端页面路由：给用户看的
- 后端接口路由：给前端程序请求的

例如：

- 页面：`/article/:slug`
- 接口：`/articles/:slug`

这两个长得相似，但职责不同。

### 2. 统一接口响应格式非常重要

如果前端统一按如下格式读取：

```json
{
  "code": 0,
  "message": "获取成功",
  "data": []
}
```

那么后端所有接口都必须保持一致。

否则会出现：

- 某些接口直接返回数组
- 某些接口返回对象包裹

最终导致前端判断逻辑混乱。

当前项目统一约定为：

- 成功：`code = 0`
- 失败：`code = 1`

### 3. 列表接口 和 详情接口 的错误语义不同

这一点非常关键。

#### 列表接口

例如：

- `/articles`
- `/articles/category/:slug`

如果没有数据，应该返回：

```json
{
  "code": 0,
  "message": "获取成功",
  "data": []
}
```

而不是抛错。

#### 详情接口

例如：

- `/articles/:slug`

如果没找到，才应该抛：

- `NotFoundException`

这体现了“资源列表”和“单资源详情”的不同语义。

## 四、开发中真实遇到的典型问题

### 1. `createdAt` 报错：明明有时间戳，为什么 TypeScript 说没有？

问题来源：

在 `Mongoose Schema` 中使用了：

```ts
@Schema({ timestamps: true })
```

运行时会自动生成：

- `createdAt`
- `updatedAt`

但如果在 `Article` 类里不声明这两个字段，TypeScript 不会知道它们存在。

正确处理方式：

```ts
createdAt: Date
updatedAt: Date
```

不需要 `@Prop()`，因为它们是自动生成字段。

### 2. 留言接口前端有数据，`messages.value` 却是空数组

原因不是 Vue 响应式失效，而是后端 `GET /messages` 返回结构不统一。

错误情况：

后端直接返回数组：

```ts
return this.messageService.findAll()
```

前端却按统一结构去拿：

```ts
result.data
```

结果：

- `result` 有值
- `result.data` 是 `undefined`
- 最终 `messages.value` 被赋成空数组

这说明：

**联调时看到“结果对象有东西”，不代表字段结构正确。**

### 3. 目录点击后页面闪到顶部再回到文章位置

问题来源：

- 目录点击时修改了 hash
- 路由层又根据 `fullPath` 重新渲染页面

结果就是：

- 页面重新挂载
- 视角闪烁

最终处理思路：

- 目录点击不让它改变页面路由状态
- 只做手动滚动定位
- 不把 hash 作为页面重新渲染的依据

### 4. 文章详情显示“这是一篇测试文章”，但前端 mock 里并没有这句话

这个问题的核心认识是：

**文章详情页已经接了真实接口，不再使用前端 mock。**

所以：

- 页面显示的正文来自后端 `content`
- 摘要卡来自后端 `summary`

如果正文出现“测试文章”，优先排查后端数据库或接口返回，而不是先怀疑前端渲染。

### 5. `category/:slug` 必须写在 `:slug` 前面

在控制器中：

```ts
@Get('category/:slug')
@Get(':slug')
```

顺序不能反。

否则：

```ts
/articles/category/technology
```

会被当成：

```ts
slug = 'category'
```

这是典型的路由匹配优先级问题。

## 五、当前项目里对前端工程结构的认识

### 1. 页面不应该承担太多业务逻辑

页面的职责应该主要是：

- 组织布局
- 调用组件
- 渲染状态

不应该把大量“数据处理逻辑”都写在页面里。

### 2. 什么逻辑应该抽成 composable

以下这类逻辑就适合抽成 composable：

- 文章列表加载：`useArticles`
- 文章详情、相关文章、上下篇：`useArticleDetail`
- 分类文章：`useCategory`
- 归档分组：`useArchive`
- 留言列表：`useGuestbook`

这样做的意义是：

- 页面只管展示
- 以后从 mock 切换到 NestJS，只改 composable 即可

### 3. 什么适合抽成组件

适合重复复用、结构稳定的 UI：

- 统一大图横幅：`PageHero`
- 文章详情头图：`ArticleHero`
- 右侧个人卡：`ProfileSidebarCard`
- 右侧统计卡：`SiteStatsCard`
- 文章目录：`ArticleToc`

这里要注意一个工程经验：

**不要为了统一命名而破坏已有组件职责。**

例如：

- 原本 `PostCard` 已经承担了文章卡片展示职责
- 那么统一 `article` 命名时，不应该随手新造重复组件把它替掉

更稳的做法是：

- 保留组件职责
- 只统一数据命名与接口语义

## 六、当前技术栈下应该优先实现的后端能力

结合当前项目目标，后端优先级建议如下：

### 第一阶段：文章系统

先完成：

- `GET /articles`
- `GET /articles/:slug`
- `GET /articles/category/:slug`
- `GET /articles/archive/list`
- `POST /articles`

这些接口打通后，前端最核心的：

- 博客页
- 文章页
- 分类页
- 归档页

就都可以走真实数据。

### 第二阶段：留言板

再完成：

- `GET /messages`
- `POST /messages`

先做平铺留言，不急着做嵌套回复。

### 第三阶段：后台能力

再补：

- `PATCH /articles/:id`
- `DELETE /articles/:id`
- 登录与 JWT

## 七、从 0 开始学习 NestJS 时，真正需要掌握的重点

当前阶段最重要的不是背概念，而是先会下面这些：

### 1. `Module / Controller / Service`

这是 NestJS 最基础的分层结构。

### 2. `Schema + Mongoose`

知道：

- 怎么定义文档结构
- 怎么查列表
- 怎么查单条
- 怎么创建数据

### 3. `DTO + ValidationPipe`

知道：

- DTO 是拿来约束请求体的
- `ValidationPipe` 开启后，DTO 才会真正生效

### 4. 前后端约定一致性

最典型的就是：

- 路由命名
- 响应结构
- 字段名

如果这三件事不统一，联调会非常痛苦。
