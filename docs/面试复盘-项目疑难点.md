# 面试复盘：项目疑难点清单

这份清单整理的是在 CoBlog 项目开发过程中，真实遇到过、讨论过、修过的问题。后面复盘时，不只是记“结论”，更重要的是理解：问题为什么会出现、我最后是怎么定位和解决的。

---

## 1. 前端数据流与响应式

### 1.1 为什么后端数据变了，前端页面不一定立刻更新
- 共享 `composable` 里的状态可能被缓存。
- 页面如果复用了同一个响应式源，但没有重新请求数据，就会继续显示旧数据。
- 解决方式：
  - 明确哪些数据需要共享。
  - 在新增、编辑、删除后主动刷新共享数据。
  - 不要为了“少请求一次”把实时性搞没了。

### 1.2 `computed + filter + includes` 这种链式逻辑到底返回什么
- `filter` 的回调返回的是 `true / false`。
- 外层 `computed` 返回的是筛选后的数组。
- 这类问题很适合面试时讲“JavaScript 数组方法 + Vue 计算属性”一起理解。

### 1.3 为什么 `...article.tags` 参与搜索，但页面没显示标签
- 搜索字段和展示字段不是一回事。
- `...article.tags` 只是参与拼接搜索文本，不代表模板一定渲染了标签列。
- 这说明：
  - 读代码时要区分“数据参与逻辑”还是“数据参与 UI”。

### 1.4 什么时候该用共享 `ref`，什么时候不该缓存
- 适合共享：文章列表、分类列表、标签列表这类全站公共数据。
- 不适合盲目缓存：后台管理页的实时数据。
- 面试可讲：
  - 共享状态提升复用性。
  - 但缓存策略要服务于业务实时性。

## 2. API 层与页面层职责

### 2.1 为什么 API 层应该只做 API
- API 层职责是：发请求、收响应。
- 如果把业务转换、URL 拼接、页面级判断都塞到 API 层，会让可读性变差。
- 这次项目里典型例子就是图片上传接口：
  - API 层只负责上传文件。
  - 页面层自己决定怎么拼完整 URL。

### 2.2 为什么“请求成功”不等于“页面业务正确”
- 有些问题不在请求层，而在页面拿到数据后的处理。
- 例如：
  - 上传接口成功了。
  - 但图片地址是相对路径。
  - 预览页运行在另一个端口，就会显示失败。

## 3. Markdown 编辑器与渲染链路

### 3.1 `textarea` 和 `md-editor-v3` 的差别是什么
- `textarea` 只是纯输入框。
- `md-editor-v3` 自带：
  - Markdown 工具栏
  - 预览
  - 图片上传回调
  - 代码块编辑体验
- 这体现了：
  - 功能复杂到一定程度，应该引成熟库，而不是继续手写。

### 3.2 Markdown 图片为什么能即时显示
- 编辑器里插入的是图片 URL。
- 预览时 Markdown 被转成 HTML：
  - `![alt](url)` -> `<img src="url">`
- 浏览器再去请求这个 `url`。

### 3.3 相对图片路径和完整 URL 的区别
- 相对路径：`/uploads/images/a.png`
- 完整路径：`http://localhost:3000/uploads/images/a.png`
- 如果前端运行在 `5173`，相对路径会默认指向前端域名。
- 所以跨端口预览时，很多场景需要在页面层补成完整地址。

### 3.4 Markdown 最终是怎么渲染的
- 先用解析器把 Markdown 转成 HTML。
- 再把 HTML 注入渲染组件。
- 如果有代码块，再配合代码高亮。
- 如果有图片路径规则，还要在渲染前或渲染后做路径补全。

## 4. 上传图片链路

### 4.1 前端上传、后端接收、静态访问是怎么串起来的
- 前端：上传本地文件。
- 后端：接收文件并保存到磁盘。
- 后端：返回图片 URL。
- 前端：把 URL 插入 Markdown，或用于封面图。
- 浏览器：通过 URL 访问静态资源。

### 4.2 为什么图片存在后端文件夹里，前端还能显示
- 因为后端做了静态资源托管。
- 文件物理上在：
  - `server/uploads/images`
- 访问地址却是：
  - `/uploads/images/...`

### 4.3 为什么上传模块后来要考虑 `schema`
- 不建 `schema` 也能上传。
- 但如果要做：
  - 素材管理
  - 图片清理
  - 封面复用
  - 上传记录查询
  就需要把上传文件信息入库。

## 5. 后台分类与标签管理

### 5.1 分类和标签到底应该以谁为准
- 不是文章自动汇总优先。
- 应该以 taxonomy 集合为准：
  - 分类来自 `categories`
  - 标签来自 `tags`
- 文章只是“使用这些 taxonomy 的内容载体”。

### 5.2 为什么删标签时不仅要删标签表，还要改文章表
- 因为文章里也存了一份 `tags` 数组。
- 如果只删 `tags` 集合，不删文章上的标签：
  - 前端文章还能显示旧标签
  - 后台搜索也还能搜到

### 5.3 标签删除后搜索还命中，为什么不一定是删除失败
- 后台搜索是综合搜索。
- 它可能匹配：
  - 标题
  - 分类
  - slug
  - 标签
- 所以“搜到了某词”不代表标签还在，可能只是标题或 slug 里还有这个词。

### 5.4 为什么更新分类/标签时要同步文章
- 分类变了：文章里的 `category` 和 `categorySlug` 也要变。
- 标签变了：文章 `tags` 数组里对应值也要变。
- 否则前后端数据会分裂。

## 6. 后端 NestJS 模块化理解

### 6.1 为什么这个项目适合 `controller + service + module (+ schema)`
- `controller`：暴露接口。
- `service`：处理业务逻辑。
- `module`：组织能力边界。
- `schema`：定义数据库结构。
- 这种结构适合中后台项目持续增长。

### 6.2 DTO、Schema、Service 分别负责什么
- DTO：校验输入数据。
- Schema：定义数据库字段。
- Service：做真正的业务处理。
- 面试时这是 NestJS 项目的高频问题。

### 6.3 为什么后端返回结构要统一
- 前端更容易统一处理成功/失败。
- 不会有的接口返回数组，有的返回对象，有的返回裸数据。
- 统一成 `ApiResponse.success(data, message)` 之后，前后端协作成本更低。

## 7. 样式与组件覆盖

### 7.1 为什么 placeholder 颜色改了像没改
- `Element Plus` 自身有一套变量和样式。
- 浏览器对 placeholder 还有自己的前缀规则。
- 再叠加全局暗色主题样式，局部样式容易被覆盖。
- 所以要同时处理：
  - 普通 placeholder
  - webkit placeholder
  - moz placeholder
  - 组件变量

### 7.2 为什么背景层只有左边能 hover
- 不一定是 hover 写错。
- 很可能是透明内容层挡住了鼠标事件。
- 这种问题本质是：
  - `z-index`
  - `pointer-events`
  - 覆盖层层级

### 7.3 为什么一个页面里“背景层 / 内容层 / 交互层”容易互相挡住
- 因为视觉层和交互层不总是一致。
- 看起来透明，不代表不会挡住鼠标。
- 这是复杂页面里很常见的真实问题。

## 8. 图表与第三方库

### 8.1 ECharts 为什么报“already initialized on the dom”
- 因为同一个 DOM 被重复 `echarts.init()`。
- 正确做法：
  - 先 `getInstanceByDom()`
  - 没有再初始化
  - 组件销毁时 `dispose()`

### 8.2 什么时候该引库，什么时候该自己写
- 业务复杂、编辑器、图表、滚动组件这种：优先引库。
- 高度视觉定制、组件库解决不了表达的时候，再自己写外观层。
- 例如：
  - `md-editor-v3` 做编辑器底层
  - `vue3-marquee` 做滚动底层
  - 外观再自己定制

## 9. 路由与页面壳子

### 9.1 独立页面和主站页面有什么区别
- 独立页：不走全站 header/footer/layout。
- 主站页：复用全站壳子。
- `About` 页面这次就反复验证了这个问题。

### 9.2 为什么同一个页面有时适合独立壳子，有时适合回到主站
- 看它承担的是：
  - 沉浸式展示
  - 还是站内普通信息页
- 本质是产品定位，不只是技术实现。

## 10. 代码清理与工程节奏

### 10.1 什么时候适合边开发边清理，什么时候适合最后统一清理
- 边开发边清：删掉明显死代码。
- 接近收尾再统一清：做系统性整理。
- 不建议一路堆到最后一次性大扫除。

### 10.2 怎么判断“死代码”
- 先查引用。
- 再确认是否是运行时、模板、动态调用。
- `script setup`、router、动态组件、第三方约定这些都要小心误判。

## 11. 面试时怎么讲

建议把这些问题归成 4 类来讲：

### 11.1 前端数据流
- 缓存策略
- 页面刷新与共享状态
- 计算属性和搜索逻辑

### 11.2 后台管理设计
- taxonomy 与文章关系
- 删除/编辑后的同步策略
- 草稿、发布、上传的管理闭环

### 11.3 上传与 Markdown 链路
- 本地图片上传
- 返回 URL
- 插入 Markdown
- 预览与前台渲染一致性

### 11.4 组件与工程规范
- 类型分层
- API 层职责
- 布局壳子拆分
- 死代码清理策略
