# CoBlog 从 0 到 1 项目落地流程

这份文档记录了当前仓库从本地开发到 Linux 线上部署的完整落地方案，同时把本次实际踩过的坑一并整理下来，后续再接手时可以直接按这份文档操作。

## 1. 项目现状

当前项目已经切成 monorepo 结构：

```text
.
├── apps/
│   ├── client/                Vue 3 + Vite 前端
│   └── server/                NestJS + MongoDB 后端
├── deploy/
│   └── nginx/
│       └── coblog.conf        Linux Nginx 配置模板
├── docs/
└── .github/workflows/
    └── deploy.yml             GitHub Actions 自动部署
```

当前关键文件：

- 前端接口地址配置：`apps/client/src/config/http.ts`
- 前端 env 读取范围：`apps/client/vite.config.ts`
- 后端 env 加载入口：`apps/server/src/app.module.ts`
- 后端启动入口：`apps/server/src/main.ts`
- Linux 部署流程：`.github/workflows/deploy.yml`
- Nginx 配置模板：`deploy/nginx/coblog.conf`

## 2. 技术选型结论

### 2.1 为什么最终选 Linux ECS

这次实践里，Windows ECS 虽然能跑，但部署链路非常别扭：

- OpenSSH 权限和公钥文件路径排查成本高
- Nginx + PM2 + Node + MongoDB 在 Windows 上维护体验差
- GitHub Actions 远程部署、目录路径、服务管理都不如 Linux 顺手

因此最终落地建议是：

- 前端：Linux ECS + Nginx
- 后端：Linux ECS + PM2
- 数据库：Linux 本机 MongoDB（当前方案）或后续迁到独立云数据库

### 2.2 当前线上约定

- 部署根目录：`/opt/coblog`
- 前端静态目录：`/opt/coblog/client/dist`
- 后端运行目录：`/opt/coblog/server`
- Nginx 配置目录：`/etc/nginx/conf.d/`
- 后端监听端口：`3000`
- MongoDB 监听：`127.0.0.1:27017`

## 3. 本地开发环境搭建

### 3.1 依赖安装

在仓库根目录执行：

```bash
pnpm install
```

如果本地切分支、构建、缓存混乱，推荐先清理再重装：

```bash
rm -rf node_modules apps/client/node_modules apps/server/node_modules .turbo
pnpm install
```

Windows 下可以手动删除这些目录后再执行 `pnpm install`。

### 3.2 本地开发环境变量

根目录手动创建：

```text
.env.development.local
```

建议内容：

```env
VITE_API_BASE_URL=http://localhost:3000
MONGODB_URI=mongodb://127.0.0.1:27017/coblog
PORT=3000
JWT_SECRET=本地开发用随机字符串
JWT_EXPIRES_IN=1d
JWT_REFRESH_SECRET=本地开发用随机字符串
MINIMAX_API_KEY=你的本地调试 key
MINIMAX_API_URL=https://api.minimax.chat/v1/text/chatcompletion_v2
MINIMAX_MODEL=MiniMax-M2.7
```

说明：

- `VITE_API_BASE_URL` 是前端用来决定“请求发给哪个后端”的变量
- `MONGODB_URI`、`PORT`、`JWT_SECRET` 这些是后端自己读取的变量

### 3.3 为什么前端用 `import.meta.env`，后端用 `process.env`

- 前端运行在浏览器，由 Vite 在构建/启动时注入变量，所以用 `import.meta.env`
- 后端运行在 Node.js 里，天然读取运行时环境变量，所以用 `process.env`

### 3.4 为什么之前本地接口会返回前端入口页

这次踩到的一个典型坑：

- 你把 `.env.development.local` 放在了仓库根目录
- 前端项目实际在 `apps/client`
- 如果 `apps/client/vite.config.ts` 没有配置 `envDir: '../..'`
- 那前端根本读不到根目录的 `VITE_API_BASE_URL`
- 结果前端请求会错误打到 Vite 自己的开发服务器，而不是 `localhost:3000`

当前仓库已经修复这个问题：

- `apps/client/vite.config.ts` 已配置 `envDir: '../..'`
- `apps/client/src/config/http.ts` 走 `VITE_API_BASE_URL`

### 3.5 本地启动命令

根目录执行：

```bash
pnpm dev:server
pnpm dev:client
```

当前端本地接口正常时：

- 页面：`http://localhost:5173`
- 接口：`http://localhost:3000`

### 3.6 本地构建验证

```bash
pnpm build:client
pnpm build:server
```

只要这两条都过，本地依赖通常就没问题。

## 4. Linux 服务器初始化

### 4.1 服务器安装项

服务器需要具备这些环境：

- Nginx
- Node.js
- pnpm
- PM2
- MongoDB

当前线上 Linux 机已经装好这些基础环境。

### 4.2 线上环境变量

服务器创建：

```text
/opt/coblog/.env
```

典型内容：

```env
MONGODB_URI=mongodb://127.0.0.1:27017/coblog
PORT=3000
JWT_SECRET=线上随机密钥
JWT_EXPIRES_IN=1d
JWT_REFRESH_SECRET=线上随机密钥
MINIMAX_API_KEY=线上真实 key
MINIMAX_API_URL=https://api.minimax.chat/v1/text/chatcompletion_v2
MINIMAX_MODEL=MiniMax-M2.7
```

注意：

- 本地 `.env.development.local` 和线上 `/opt/coblog/.env` 是两套独立配置
- 不要把线上真实密钥写回仓库

### 4.3 后端为何默认跑 3000

`apps/server/src/main.ts` 当前逻辑：

```ts
await app.listen(process.env.PORT ?? 3000)
```

所以：

- 本地开发时后端默认监听 `3000`
- 线上也是监听 `3000`
- 浏览器不会直接访问这个端口，而是由 Nginx 反代

## 5. Nginx 配置与线上访问路径

当前模板文件：`deploy/nginx/coblog.conf`

核心职责分两部分：

### 5.1 托管前端静态资源

```nginx
root /opt/coblog/client/dist;
index index.html;

location / {
  try_files $uri $uri/ /index.html;
}
```

说明：

- 前端使用 Vue Router history 模式
- 必须有 `try_files ... /index.html`
- 否则刷新详情页、分类页、文章页会 404

### 5.2 反代后端 API

```nginx
location ~ ^/(articles|auth|messages|taxonomies|uploads|ai)(/.*)?$ {
  proxy_pass http://127.0.0.1:3000;
}
```

说明：

- 线上浏览器只访问 `http://域名或公网IP`
- 后端真实服务在 `127.0.0.1:3000`
- Nginx 负责把接口请求转发给 Nest

### 5.3 验证 Nginx

```bash
nginx -t
systemctl reload nginx
systemctl status nginx
```

## 6. GitHub Actions 自动部署流程

当前工作流：`.github/workflows/deploy.yml`

### 6.1 部署触发方式

- push 到 `main` 自动触发

### 6.2 GitHub Secrets

需要配置：

- `ECS_HOST`
- `ECS_USER`
- `ECS_SSH_KEY`

当前 Linux 机部署建议：

- `ECS_HOST=8.136.8.240`
- `ECS_USER=root`
- `ECS_SSH_KEY=当前 Linux 机对应私钥全文`

### 6.3 为什么 GitHub 要放私钥，服务器放公钥

SSH 认证规则始终不变：

- 客户端持有私钥
- 服务端保存公钥

所以：

- GitHub Actions 主动连接服务器，GitHub 需要私钥
- Linux 服务器被连接，Linux 上保存对应公钥

Windows 和 Linux 规则一样，区别只是公钥文件位置和权限处理不同。

### 6.4 工作流实际做了什么

1. checkout 仓库
2. `pnpm install --frozen-lockfile`
3. `pnpm build`
4. 生成部署用后端 `package.json`
5. 配置 SSH
6. 验证远程登录
7. 上传：
   - 前端 `dist`
   - 后端 `dist`
   - 部署用 `package.json`
   - Nginx 配置
8. 远端执行：
   - `npm install --omit=dev`
   - `pm2 restart/start`
   - `pm2 save`
   - `nginx -t`
   - `systemctl reload nginx`

### 6.5 为什么部署时生成临时后端 `package.json`

后端源码里保留：

```json
"@coblog/types": "workspace:*"
```

这是源码开发需要的共享类型依赖。  
但远端部署只传后端应用，不带整个 workspace，所以部署时必须临时去掉这项，否则远端 `npm install` 会失败。

当前工作流已经用 `.deploy/server-package.json` 处理了这个问题。

## 7. 数据库落地与迁移

### 7.1 当前数据库方案

当前线上 Linux 使用本机 MongoDB：

```env
MONGODB_URI=mongodb://127.0.0.1:27017/coblog
```

优点：

- 配置最简单
- 本机服务直连

缺点：

- 服务器释放后，数据库可能一起没
- 必须自己做备份

### 7.2 首次把本地 Mongo 数据迁到 Linux

#### 本地导出

Windows 本机：

```powershell
& "C:\Program Files\MongoDB\Tools\100\bin\mongodump.exe" --uri="mongodb://127.0.0.1:27017/coblog" --out "$env:USERPROFILE\Desktop\mongo-backup"
```

#### 上传到 Linux

```powershell
scp -i C:\Users\123\.ssh\zsh.pem -r "$env:USERPROFILE\Desktop\mongo-backup" root@8.136.8.240:/opt/coblog/
```

#### 服务器恢复

```bash
mongorestore --drop --db coblog /opt/coblog/mongo-backup/coblog
```

#### 恢复后重启后端

```bash
pm2 restart coblog-api
pm2 logs coblog-api --lines 50
```

## 8. 手动备份与回拉数据库

如果你不想做自动化脚本，手动拉一拉就够用：

### 8.1 服务器导出

```bash
mongodump --db coblog --out /opt/coblog/mongo-backup
```

### 8.2 本地拉回

```powershell
scp -i C:\Users\123\.ssh\zsh.pem -r root@8.136.8.240:/opt/coblog/mongo-backup "$env:USERPROFILE\Desktop\mongo-backup-from-server"
```

### 8.3 本地恢复

覆盖本地正式库：

```powershell
& "C:\Program Files\MongoDB\Tools\100\bin\mongorestore.exe" --drop --db coblog "$env:USERPROFILE\Desktop\mongo-backup-from-server\mongo-backup\coblog"
```

如果你只想先看，不覆盖正式库：

```powershell
& "C:\Program Files\MongoDB\Tools\100\bin\mongorestore.exe" --drop --db coblog_sync_tmp "$env:USERPROFILE\Desktop\mongo-backup-from-server\mongo-backup\coblog"
```

## 9. Git 分支与协作建议

### 9.1 推荐规则

最省事的方案：

- 只在 `main` 开发
- 部署只看 `main`
- `master` 不直接开发，只做镜像或直接弃用

### 9.2 为什么之前切分支会卡

这次出现过这些问题：

- `.git/index.lock` 残留
- `.github/workflows` 被 Windows 进程占用，删不掉
- `dist`、`node_modules`、`.turbo` 没被正确忽略，Source Control 一次扫出上万文件

当前 `.gitignore` 已补上 monorepo 需要的忽略项：

- `**/node_modules/`
- `**/dist/`
- `.turbo/`
- `apps/server/uploads/`

### 9.3 如果必须同步 `master`

让 `master` 跟 `main` 一样时，直接用单向镜像：

```powershell
git push origin refs/heads/main:refs/heads/master --force-with-lease
git branch -f master origin/master
```

## 10. 本次实际踩过的坑

### 10.1 Windows ECS SSH 认证排查

结论：

- 不是 SSH 机制和 Linux 不同
- 是 Windows 的公钥文件位置、权限、编码更容易踩坑
- 最终确认公钥登录是否成功，一定要用**禁用密码回退**的严格测试命令

### 10.2 `VITE_API_BASE_URL` 看起来像后端变量，其实是前端变量

它表示“前端请求打给哪个后端”。  
读取它的是前端浏览器代码，不是后端自己。

### 10.3 文章详情 500

曾经出现过文章详情接口 500，本质是后端统一序列化时对 `updatedAt` 写死，遇到不完整投影对象会炸。后续如果再碰到文章详情接口异常，优先检查：

- `apps/server/src/articles/article.service.ts`
- 文章详情接口 `/articles/:slug`
- PM2 日志里是否出现 `toISOString` / `undefined`

### 10.4 MongoDB 没启动 vs MongoDB 地址写错

只要日志里出现：

```text
ECONNREFUSED 127.0.0.1:27017
```

就先检查两件事：

1. MongoDB 服务是否真的在跑
2. `.env` 里的 `MONGODB_URI` 是否写成了错误地址
